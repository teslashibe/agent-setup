# MCP architecture

This document describes how the Model Context Protocol (MCP) layer is wired
into `agent-setup`. It covers transport, auth, registry/dispatch,
per-user agent provisioning, and the drift-prevention mechanisms that keep
each platform's tool surface in lockstep with its underlying scraper.

The companion document is [`mcp-inventory.md`](./mcp-inventory.md), which
is generated by `cmd/mcp-inventory` and lists every tool currently
registered.

## Component map

### `internal/mcp` — transport, dispatch, shaping

| File | Responsibility |
|---|---|
| `server.go` | Tool dispatcher. Holds the `Registry`, `CredentialResolver`, `ResponseShaper`. `CallTool` and `ListTools` are the only entry points. |
| `registry.go` | Maps platform ID → `PlatformBinding{Provider, NewClient, ValidateCredential}`. Frozen after construction. |
| `transport.go` | HTTP / JSON-RPC 2.0 binding. Mounts routes, parses single + batch requests, handles `initialize` / `tools/list` / `tools/call`. |
| `clientcache.go` | Per-request scratch cache so a batch of N calls to the same platform reuses one underlying client. |
| `shape.go` | `ResponseShaper` — recursively truncates strings, caps array length, then JSON-marshals; if still over `MaxResponseBytes` it falls back to a tagged truncation envelope. |

### `internal/mcp/platforms` — wiring

`platforms.All()` returns one `Plugin` per supported platform. Each plugin
provides:

- `Provider` — the scraper's `mcptool.Provider` (lives in the scraper
  module under `<scraper>/mcp`).
- `NewClient(blob json.RawMessage) (any, error)` — parses the encrypted
  credential blob into the constructor shape that scraper expects (cookie,
  token, full cookies map, etc.).
- `Validator` — implements `credentials.Validator`; used by the credential
  HTTP handler to fail fast before storing a malformed blob.

Adding a new platform = add one entry to `All()` plus a `simpleValidator`
declaration. The Cursor rule
`.cursor/rules/mcp-tool-conventions.mdc` walks through the full process.

### `internal/credentials` — encrypted credential store

| File | Responsibility |
|---|---|
| `cipher.go` | AES-GCM encrypt/decrypt with `CREDENTIALS_ENCRYPTION_KEY` (32-byte hex). |
| `service.go` | CRUD over `platform_credentials` (PG); fans out `Validator`s by platform. |
| `handler.go` | Fiber routes mounted under `/api/platforms`. See [`credentials-setup.md`](./credentials-setup.md). |

### `internal/agent` — per-user provisioning

`agent.Provisioner.EnsureForUser(ctx, userID)` returns the cached
`(agentID, environmentID)` pair for a user, lazily creating them on first
chat. Each user gets a dedicated Anthropic Agent whose only MCP server is
`https://<host>/mcp/u/<jwt>/v1` — the JWT is signed for that user, scoped
to a single subject claim, and never leaves the Agent's stored config.

### `internal/auth` — path-mounted JWT

Anthropic's MCP server config doesn't accept arbitrary auth headers, so we
embed the JWT in the URL path and validate it via
`auth.RequirePathAuth("token")`, which forwards a synthetic
`Authorization: Bearer <token>` to the existing `magiclink.AuthenticateBearer`.

## Request lifecycle

```text
POST /mcp/u/<jwt>/v1
Body: {"jsonrpc":"2.0","id":1,"method":"tools/call",
       "params":{"name":"linkedin.search_people","arguments":{...}}}

  1. RequirePathAuth(token)
       → auth.SetUser(ctx, user)
  2. Transport.handleRPC
       → splits batch, dispatches each
  3. Server.CallTool(ctx, "linkedin.search_people", args)
       → Registry.binding("linkedin")
       → CredentialResolver.Resolve(ctx, userID, "linkedin")  (PG + AES-GCM)
       → clientCache.Get("linkedin", binding.NewClient(blob)) (per-request reuse)
       → mcptool.Provider.Call(ctx, "search_people", args, client)
            → tool.InputT struct → handler(ctx, client, in)
            → returns (any, error)
       → ResponseShaper.Shape(result)
            → truncate strings @ 800 chars
            → cap array slices to limit (default 10, max 50)
            → fall back to summary envelope if > MaxResponseBytes
  4. Transport returns JSON-RPC envelope (single or batched)
```

## Response shaping (token efficiency)

Every tool result passes through `ResponseShaper.Shape`. Defaults are set
in `internal/config/config.go`:

| Knob | Default | Env var | Why |
|---|---:|---|---|
| `MaxItemsPerPage` | 50 | `MCP_MAX_ITEMS_PER_PAGE` | Hard ceiling for any array slice. |
| `MaxStringLen` | 800 | `MCP_MAX_STRING_LEN` | Per-string truncation budget; longer strings get `…` and a `_truncated_chars` sibling. |
| `MaxResponseBytes` | 65536 | `MCP_MAX_RESPONSE_BYTES` | If marshalled JSON exceeds this, the shaper retries with a smaller string budget; otherwise it returns a `{"_summary": "...", "_truncated": true}` envelope. |
| `MaxDepth` | 8 | `MCP_MAX_DEPTH` | Belt-and-suspenders against deeply nested objects. |
| `DefaultPageLimit` | 10 | `MCP_DEFAULT_PAGE_LIMIT` | Recommended default for `Limit` input fields when undefined. |

Tool authors should still:
- declare `Limit int` in their `InputT` with `jsonschema:"default=10,minimum=1,maximum=50"`,
- emit pre-shaped responses via `mcptool.PageOf(items, total, page, hasMore)`,
- pre-truncate long fields with `mcptool.TruncateString(s, 800)`.

The shaper is a safety net, not a substitute.

## Drift-prevention

Five mechanisms keep MCP and the underlying clients in sync. All five are
required — if you bypass any of them, the next CI run or audit will catch
it.

1. **Shared contract (`mcptool`)** — every scraper's `mcp/` subpackage
   imports `github.com/teslashibe/mcptool` and uses the same `Tool`,
   `Provider`, `Error` types. `agent-setup` only knows about the contract,
   not the individual scrapers.

2. **Schema-from-structs** — tool input schemas are reflected from
   `InputT` Go structs via `invopop/jsonschema`. If you add a field to
   the client method, you add it to `InputT`; the schema follows
   automatically.

3. **Coverage tests** — every scraper has a
   `mcp/coverage_test.go` containing
   `mcptool.AssertCoverage(t, Provider(), Methods, Excluded)`. CI fails if
   a public client method is added without a corresponding tool **or**
   explicit entry in `excluded.go`.

4. **Auto-bump CI** — when `agent-setup` opens a Renovate/Dependabot PR
   for a scraper version bump, CI runs `go test ./...` (which includes
   the registry composition test) and the inventory generator; both must
   pass before merge.

5. **Audit rules** — `.cursor/rules/mcp-tool-conventions.mdc` and the
   org-wide `issue-audit-user-stories.mdc` instruct any AI agent
   inspecting an MCP-related issue to re-verify (a)/(b)/(c) before
   declaring the work complete.

## Kubernetes / production notes

- **Single ingress** — the same Fiber pod serves `/api/...`, `/mcp/...`,
  and the magic-link callback. No sidecar.
- **Ingress timeouts** — bump to ≥120s; managed-agent calls can run long
  scraper jobs.
- **Egress IP** — pin Fiber pods to a stable egress (NAT gateway or
  pinned node) so LinkedIn / X anti-bot heuristics don't get tripped by
  rotating cluster IPs.
- **Secrets** — `CREDENTIALS_ENCRYPTION_KEY` and `MAGICLINK_JWT_SECRET`
  must be passed via K8s Secrets. **Never** commit them.
- **Idempotent provisioning** — `agent.Provisioner` writes the
  Anthropic agent/environment IDs to `users.anthropic_agent_id` /
  `users.anthropic_environment_id`. Concurrent first-chat requests for the
  same user are serialised by a per-user `sync.Mutex`.
- **Database pool** — the MCP layer borrows the same `pgxpool.Pool` as
  the rest of the app. Increase `DB_POOL_MAX` if you expect heavy fan-out.
- **Observability** — every tool call is logged with
  `{user_id, platform, tool, latency_ms, bytes_out, truncated}`; pipe
  these to your standard log aggregator.
